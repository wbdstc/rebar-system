# 钢筋工程智能管控平台

一个面向钢筋工程施工验收场景的智能化系统，围绕“图纸解析、现场检测、材料核验、记录留痕”建立质量管控闭环。项目采用 `Flask + Vue 3` 架构，结合 OCR、目标检测与视觉大模型能力，辅助完成钢筋工程的快速核验与合规性判断。

## 1. 项目定位

本项目主要服务于钢筋工程质量管理过程中的几个高频环节：

- 隐蔽工程验收
- 进场材料验收
- 原材表面轧印识别
- 检测记录归档与导出

系统目标是将传统依赖人工经验、手工比对、纸面留痕的验收流程，升级为更高效、更标准、更可追溯的数据化流程。

## 2. 核心能力

### 2.1 隐蔽工程验收

支持柱、梁、板、墙等常见构件的钢筋检测与验收，覆盖：

- 纵筋数量识别
- 箍筋或网片间距检测
- 设计值与现场实测值比对
- 合规状态提示

在柱、梁等场景中，系统支持先解析 CAD 或图纸截图，再将解析出的设计参数自动带入现场比对流程。

### 2.2 CAD 图纸智能解析

系统支持上传 CAD 或 PDF 截图，并通过视觉大模型提取配筋相关参数，例如：

- 柱构件角筋、中部筋、总筋数
- 梁构件上下部纵筋、腰筋、箍筋信息
- 板、墙类构件设计间距

解析结果既保留为可读的 AI 分析报告，也输出为结构化参数，便于后续自动填充检测表单。

### 2.3 进场材料验收

支持对钢筋端面图片进行检测，辅助完成：

- 数量统计
- 端面目标标注
- 参考物标定后的直径估算

适用于现场拍照后快速完成材料进场抽检。

### 2.4 原材微观识别

支持对钢筋表面轧印进行识别，自动提取：

- 钢筋牌号
- 是否带抗震标记 `E`
- 公称直径
- 原始识别文本

该能力可用于原材核验与信息登记，降低人工判读成本。

### 2.5 检测记录与报表

系统提供检测记录中心，支持：

- 检测结果入库
- 历史记录查询
- 分页筛选
- 单条记录删除
- Excel 报表导出

## 3. 业务闭环

项目的典型使用流程如下：

1. 上传设计图纸截图，系统自动解析钢筋设计参数。
2. 上传施工现场照片，系统自动完成数量、间距或直径相关检测。
3. 将现场检测结果与设计值进行比对，输出合规状态。
4. 保存检测记录并进入记录中心统一管理。
5. 按需导出 Excel 报表，用于验收留档或汇报展示。

## 4. 技术架构

### 4.1 前端

- Vue 3
- Vite
- Vue Router
- Element Plus
- Axios
- Marked

前端主要负责页面交互、图片上传、参数配置、检测结果可视化与记录管理。

### 4.2 后端

- Flask
- Flask-CORS
- Flask-SQLAlchemy
- PyMySQL
- OpenCV
- Pillow
- pytesseract
- openpyxl
- MinIO SDK

后端主要负责图像分析流程编排、模型接口调用、记录存储、报表导出与健康检查。

### 4.3 外部能力

- Roboflow：目标检测能力，用于数量与间距类识别
- 智谱 AI GLM-4V：图纸解析与轧印识别
- Tesseract OCR：图纸文字识别与结构化抽取
- MinIO：图片对象存储
- MySQL：业务数据存储

## 5. 系统模块

前端页面主要分为以下几个模块：

- `首页`：系统总入口与模块导航
- `隐蔽工程验收`：CAD 解析 + 现场检测 + 合规判定
- `进场材料验收`：端面计数与直径估算
- `原材微观检测`：钢筋轧印识别
- `检测数据中心`：记录查询与导出

后端主要接口包括：

- `POST /api/analyze`：统一检测接口
- `POST /api/parse_cad`：CAD 图纸智能解析
- `POST /api/verify_material`：钢筋轧印识别
- `POST /api/ocr`：OCR 识别
- `POST /api/check_compliance`：合规性检查
- `GET/POST /api/records`：检测记录查询与创建
- `DELETE /api/records/<id>`：删除记录
- `POST /api/export_excel`：导出报表
- `GET /api/health`：服务健康检查

## 7. 环境要求

建议开发环境如下：

- Python 3.11 及以上
- Node.js 18 及以上
- MySQL 8.x
- MinIO
- Windows 开发环境

如果需要启用 OCR，本机还需正确安装 Tesseract。

## 8. 快速启动

### 8.1 数据库初始化

先准备 MySQL 服务，然后执行：

```powershell
python init_db.py
```

该脚本会：

- 创建 `rebar_system` 数据库
- 初始化项目所需的数据表

### 8.2 启动后端

```powershell
pip install -r requirements.txt
python app.py
```

默认后端地址：

```text
http://localhost:5000
```

### 8.3 启动前端

```powershell
cd frontend
npm install
npm run dev
```

默认前端地址：

```text
http://localhost:5173
```

### 8.4 Windows 一键启动

项目提供了 `start-system.bat`，会尝试：

- 启动 MinIO
- 启动 Flask 后端
- 启动 Vue 前端

如果你的 MinIO 路径不是脚本中默认的 `D:\tool\minio.exe`，请先按实际环境修改脚本。

## 10. 数据存储说明

系统当前采用两类存储：

- MySQL：存储检测记录、结构化参数、合规结果等业务数据
- MinIO：存储原始图片或处理结果图片

当 MinIO 不可用时，后端会回退到本地临时目录保存文件。

## 11. 项目价值

本项目重点解决了以下实际问题：

- 人工数筋、量间距效率低，容易漏检或误判
- 图纸参数录入与现场核验脱节，流程不连续
- 原材轧印识别依赖经验，信息登记效率低
- 检测记录分散，后续追溯与报表整理成本高

通过引入图纸端、现场端、材料端的一体化智能分析能力，系统可以帮助项目团队实现：

- 更快的验收效率
- 更清晰的质量留痕
- 更规范的过程管理
- 更方便的成果汇报与资料归档
